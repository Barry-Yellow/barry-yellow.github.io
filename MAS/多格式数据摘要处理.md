# 多格式数据与摘要处理方案

## 核心设计原则

统一的多格式处理流程：**格式解析 → 文本提取 → 噪声过滤 → LLM 摘要 → 价值评估 → 分层存储**

---

## 各格式处理方案

### 1. PDF 研报

**处理流程**：
- 基于 PyMuPDF 逐页提取文本与表格
- 页面高度 8% 阈值 + 正则匹配（"第X页"、公司 logo）双重过滤页眉页脚
- 表格识别：按 Y 坐标聚类为行，按 X 坐标对齐为列，构建 Markdown 表格
- 输出：纯文本 + 结构化表格

**特点**：长篇幅、多页、含复杂表格，处理成本高但信息价值通常较高

---

### 2. 图片（K线图、财务图表、截图）

**处理流程**：
- 图像预处理：二值化、去噪、对比度增强（针对低质量截图）
- 图表类型识别：K线图 → 提取 OHLC 数据；表格图 → OCR 识别；其他 → 通用 OCR
- 使用 PaddleOCR（中文优化）或 Tesseract 进行 OCR，confidence > 0.7 的结果保留
- 输出：结构化数据（若是表格/K线）或 OCR 文本

**特点**：信息密度高但格式多样，OCR 精度依赖图片质量，需要置信度过滤

---

### 3. HTML 网页（新闻、资讯、公告）

**处理流程**：
- 移除脚本、样式、导航、广告等噪声标签（`<script>`, `<style>`, `<nav>`, `<aside>` 等）
- 优先定位 `<main>` 或 `<article>` 正文区域，若无则按内容密度启发式选择
- 保留表格（`<table>`）和列表（`<ul>`, `<ol>`）结构，转为 Markdown
- 提取元数据：`<title>`, `<meta name="description">`, `<time>` 等
- 输出：纯文本 + 元数据

**特点**：噪声多、结构复杂，但解析相对稳定，处理成本低

---

### 4. JSON 结构化数据（API 返回、数据库导出）

**处理流程**：
- 直接解析 JSON，提取关键字段：`title`, `content`, `summary`, `date`, `source`, `author`
- 若为嵌套结构，按优先级展平：优先取 `content` > `body` > `text` > `description`
- 若包含数组（如多条新闻），逐条处理
- 数据验证：检查必填字段，缺失则标记为 `incomplete`
- 输出：标准化字段集合

**特点**：格式规范、处理简单，但字段命名不统一需要适配

---

### 5. 日志文件（交易日志、系统日志、爬虫日志）

**处理流程**：
- 行式解析，按分隔符（空格、制表符、JSON）拆分
- 时间戳提取与标准化（转为 ISO 8601 格式）
- 关键字段提取：`level` (ERROR/WARN/INFO), `message`, `error_code`, `stack_trace`
- 日志聚合：相同 error_code 的日志合并，统计出现次数
- 异常检测：ERROR 级别日志标记为 `high_priority`，WARN 标记为 `medium_priority`
- 输出：聚合后的日志摘要 + 异常列表

**特点**：数据量大、格式多样，需要聚合和优先级排序

---

### 6. 音频（财报电话会议、分析师访谈）

**处理流程**：
- 长音频分段处理：按 30 秒或 1 分钟分割，避免单次转录超时
- 语音转文本：使用 OpenAI Whisper 或 PaddleSpeech，输出带时间戳的转录文本
- 说话人识别（可选）：区分不同发言人，标记为 `[Speaker A]`, `[Speaker B]`
- 文本后处理：去除重复词、修复转录错误（如数字识别错误）
- 分段摘要：每 5 分钟生成一个小摘要，最后合并为全文摘要
- 输出：带时间戳的转录文本 + 分段摘要

**特点**：处理成本最高（需要 STT 模型），但信息价值高，适合重要会议

## 统一的摘要与价值评估流程

### 摘要生成

所有格式的文本提取完成后，统一调用 LLM 生成摘要：

```
Prompt 模板：
  你是金融分析助手。
  原始内容：{extracted_text[:2000]}
  
  请生成一份 200-300 字的摘要，突出：
  1. 核心观点或关键数据
  2. 时间信息（如有）
  3. 涉及的标的或板块
  
  输出格式：
  摘要：{summary}
  关键词：{keywords}
  信息价值：{value_level}
```

### 价值评估

LLM 同时评估信息价值（high/medium/low）：

| 价值等级 | 判断标准 | 存储策略 |
|---------|---------|---------|
| **high** | 重要政策、重大事件、深度分析、数据驱动结论 | 长期记忆 + RAG 向量库 |
| **medium** | 普通研报、行业资讯、常规数据 | RAG 向量库 |
| **low** | 日常快讯、重复信息、低相关性内容 | 短期会话（TTL=24h） |

---

## 分层存储策略

```
三层记忆架构：

短期会话（Session Memory）
  ├─ 存储：内存 + 临时文件（TTL=24h）
  ├─ 用途：当前对话上下文、低价值快讯
  └─ 特点：快速访问，自动过期

临时文件（Temporary Storage）
  ├─ 存储：本地文件系统（data/temp/）
  ├─ 用途：处理中间结果、原始提取文本
  └─ 特点：便于调试，定期清理

RAG 向量库（ChromaDB）
  ├─ 存储：向量数据库
  ├─ 用途：中长期知识检索
  ├─ 入库条件：value_level >= medium
  └─ 特点：支持语义检索，支持日期过滤
```

## 多格式处理的完整数据流

```
用户输入 "帮我分析最近新能源板块的研报和新闻"
  │
  ├─ 数据采集 Agent
  │   ├─ 爬取上交所研报 PDF → PDF 处理 → 摘要 + 价值评估
  │   ├─ 爬取财联社资讯 HTML → HTML 清洗 → 摘要 + 价值评估
  │   ├─ 调用东方财富 API JSON → JSON 解析 → 摘要 + 价值评估
  │   └─ 收集交易日志 → 日志聚合 → 异常摘要 + 价值评估
  │
  ├─ 摘要 Agent
  │   ├─ 按价值分层存储
  │   ├─ high → 长期记忆 + RAG
  │   ├─ medium → RAG
  │   └─ low → 短期会话
  │
  ├─ Agentic RAG
  │   ├─ 查询改写："新能源 锂电池 板块 主力资金 最近一周"
  │   ├─ 混合检索（向量 + BM25）
  │   ├─ 日期过滤（最近 7 天）
  │   └─ 充分性评估 → 多轮迭代
  │
  └─ 分析 Agent
      └─ 基于检索结果 + 原始数据生成分析
```

## 关键参数配置

```yaml
multi_format_processing:
  pdf:
    header_footer_threshold: 0.08  # 页面高度 8%
    table_detection: true
    
  image:
    ocr_confidence_threshold: 0.7
    image_preprocessing: true  # 二值化、去噪
    chart_type_detection: true  # K线图、表格识别
    
  html:
    noise_tags: [script, style, nav, aside, footer]
    content_area_priority: [main, article, .content, #main]
    
  json:
    key_priority: [content, body, text, description]
    required_fields: [title, date]
    
  log:
    aggregation_window: 300  # 5 分钟聚合一次
    error_priority_threshold: ERROR
    
  audio:
    segment_duration: 60  # 1 分钟分段
    speaker_diarization: true
    timestamp_alignment: true
    
  summarization:
    max_length: 300  # 字符数
    temperature: 0.3  # 摘要需要稳定
    
  memory:
    short_term_ttl_hours: 24
    long_term_max_entries: 1000
```

## 处理成本与优先级

| 格式 | 处理成本 | 信息价值 | 优先级 | 建议 |
|------|---------|---------|--------|------|
| JSON | 低 | 中 | 高 | 实时处理 |
| HTML | 低 | 中 | 高 | 实时处理 |
| 日志 | 低 | 低 | 中 | 定时聚合 |
| PDF | 中 | 高 | 高 | 异步处理 |
| 图片 | 中 | 中 | 中 | 按需处理 |
| 音频 | 高 | 高 | 低 | 离线处理 |

**建议策略**：
- 低成本格式（JSON/HTML/日志）：实时处理，立即入库
- 中成本格式（PDF/图片）：异步队列处理，批量入库
- 高成本格式（音频）：离线处理，仅处理重要会议
