📋 总览：多智能体框架对比

多智能体系统（Multi-Agent System, MAS）通过多个AI代理的协作，能够完成复杂任务，比单体LLM具有更强的推理能力、任务分工灵活性和容错性。本文档深入分析五个主流多智能体框架的架构设计与实现细节。

框架 组织架构 核心设计 记忆机制 通信方式 适用场景
Magentic-One Orchestrator + 4 Specialists 主编排器驱动任务分解 共享账本(Ledger) 消息传递 复杂网页/文件任务
CrewAI Crew + Agents + Tasks 角色扮演团队协作 记忆模块 顺序/层次流程 研究、分析、内容创作
OpenAI Swarm Agent + Handoffs 极简交接机制 上下文继承 函数返回转交 轻量级路由场景
MetaGPT SOP驱动软件公司 标准作业程序嵌入 结构化团队记忆 SOP定义的消息流 软件开发生成
MAR 图结构 + 决策节点 空间/时间边优化 拓扑感知的上下文 空间+时间消息传递 数学/推理任务

1. 框架整体架构

Magentic-One是微软发布的开源多智能体系统，基于AutoGen框架构建，采用Orchestrator-Managed（主编排器）架构。系统由一个主编排器（Orchestrator）和四个专业智能体组成，负责解决复杂的网页和文件操作任务。

Magentic-One 架构分层

Orchestrator
↓
File Surfer
Web Surfer
Computer Terminal
Coder
↓
File System
Web Browser
Terminal
Python Executor
↓
Shared Ledger

2. 内置核心智能体角色与职责

Magentic-One定义了四个专业智能体和一个主编排器：

🧭 Orchestrator（主编排器）

负责任务分解、协调子智能体工作、管理共享账本。是系统的"大脑"，决定任务分配和执行顺序。

📄 File Surfer（文件导航员）

专门处理本地文件操作，包括读取、搜索、分析文档内容。

🌐 Web Surfer（网页导航员）

专门处理网页浏览任务，包括搜索、点击、填表、内容提取。

💻 Computer Terminal（终端操作员）

执行命令行操作，处理代码编译、脚本运行等终端任务。

⌨️ Coder（编码员）

负责编写和调试代码，处理编程相关任务。

3. 智能体之间如何交互

Orchestrator通过指令-响应模式与专业智能体交互：

1
任务接收：用户向Orchestrator提交复杂任务
2
任务分解：Orchestrator将任务分解为子任务队列
3
指派执行：向相应专业智能体发送执行指令
4
结果收集：收集各智能体执行结果
5
状态更新：将结果写入共享账本，更新任务进度

4. 消息传递机制与路由规则

Magentic-One采用共享账本（Ledger）作为消息传递的核心媒介：

  • 所有智能体通过读写共享账本进行通信
  • 账本记录任务状态、中间结果、待完成事项
  • Orchestrator负责维护账本的一致性
  • 智能体通过订阅特定账本条目获取相关信息

5. 记忆/上下文共享方案

共享账本系统包含三个主要部分：

📝 任务清单（Task List）

记录所有待执行的子任务及其状态

📊 进度跟踪（Progress Tracker）

记录已完成任务的结果和当前进展

📋 事实记录（Factual Notes）

存储从网页、文件中提取的关键信息

6. 任务规划（Plan）机制

Orchestrator采用迭代式任务分解：

// 伪代码示例
class Orchestrator:
    def plan(self, task):
        # 1. 理解任务目标
        subtasks = self.decompose(task)
        
        # 2. 检查依赖关系
        ready_tasks = self.get_ready_tasks(subtasks)
        
        # 3. 分配给专业智能体
        for subtask in ready_tasks:
            agent = self.select_agent(subtask)
            result = agent.execute(subtask)
            
        # 4. 评估结果，决定是否需要重规划
        if not self.is_satisfied(result):
            remaining = self.replan(result)
            # 继续执行剩余任务

7. 工作流编排方式

Magentic-One采用动态工作流：

  • 任务不是预定义的流程，而是动态生成和调度
  • Orchestrator根据当前状态决定下一步行动
  • 支持条件分支和循环，直到任务完成
  • 提供终止条件检测，自动结束任务

8. 核心系统提示词模板

Orchestrator System Prompt（核心指令）

你是一个任务编排器，负责协调多个专业智能体完成复杂任务。

你的职责：
1. 将复杂任务分解为可执行的子任务
2. 选择合适的专业智能体执行每个子任务
3. 监控任务进度，管理共享账本
4. 处理错误并进行必要的重规划

工作流程：
1. 分析用户请求，确定任务目标
2. 列出完成目标所需的步骤
3. 为每个步骤选择最合适的智能体
4. 收集结果并更新任务状态
5. 如遇失败，尝试重新规划路径

重要原则：
- 保持任务状态的准确性
- 确保智能体之间的信息共享
- 最小化不必要的交互
- 及时识别并处理错误

9. 优缺点与适用场景

✅ 优点

  • 透明可控：共享账本提供完整的任务追踪
  • 任务分解能力强：Orchestrator擅长处理复杂多步骤任务
  • 人机协作友好：支持人类介入和监督
  • 基于成熟AutoGen：有强大的社区支持

❌ 缺点

  • 资源消耗高：主编排器可能成为瓶颈
  • 延迟较大：多智能体协调带来额外开销
  • 配置复杂：需要为不同任务调整智能体组合
  • 专业性受限：4个固定智能体类型可能不够灵活

适用场景：复杂网页自动化、数据采集与分析、文件处理流水线、需要人类监督的多步骤任务

1. 框架整体架构

CrewAI是一个专注于角色扮演团队协作的多智能体框架，模拟真实团队的分工合作模式。其核心理念是"招聘-分配-执行"，让每个AI智能体像真实员工一样工作。

CrewAI 架构分层

Crew（团队）
↓
Agent 1
Agent 2
Agent N...
↓
Task 1
Task 2
Task N...
↓
Memory | Knowledge | Tools

2. 内置核心智能体角色与职责

CrewAI采用开放式角色定义，用户可以灵活定义各种角色：

👤 Agent（智能体）

核心组成：role（角色）、goal（目标）、backstory（背景故事）

每个Agent可以绑定多个Tools，具备特定领域的专业知识

📋 Task（任务）

描述、期望输出、关联Agent、上下文依赖

👥 Crew（团队）

Agent集合 + Task集合 + Process（顺序/层次）

3. 智能体交互流程

CrewAI支持两种主要流程：

1
Sequential（顺序执行）：按任务定义顺序执行，上游任务输出可作为下游输入
2
Hierarchical（层次执行）：类似管理结构，有Manager协调任务的分配和结果整合

4. 消息传递机制

CrewAI通过上下文注入（Context）实现消息传递：

# CrewAI 核心代码示例
from crewai import Agent, Task, Crew, Process

# 定义研究员
researcher = Agent(
    role='高级研究员',
    goal='发现AI领域的最新突破',
    backstory='你是一个对技术趋势有敏锐嗅觉的专家',
    tools=[search_tool]
)

# 定义作家
writer = Agent(
    role='内容作家',
    goal='创建易于理解的科普文章',
    backstory='你擅长将复杂技术翻译成通俗语言',
    tools=[None]
)

# 定义任务
task1 = Task(
    description='研究2025年AI Agent的最新趋势',
    expected_output='详细的研究报告',
    agent=researcher
)

task2 = Task(
    description='基于研究报告创作文章',
    expected_output='科普文章初稿',
    agent=writer,
    context=[task1]  # 依赖前一个任务
)

# 创建团队并启动
crew = Crew(
    agents=[researcher, writer],
    tasks=[task1, task2],
    process=Process.sequential
)

result = crew.kickoff()

CrewAI提供三层记忆系统：

1. 框架整体架构

OpenAI Swarm是OpenAI官方推出的轻量级多智能体研究框架，核心设计哲学是"极简主义"。它只有两个核心概念：Agent（智能体）和Handoffs（交接），却实现了灵活的多智能体协作。

Swarm 核心概念

Agent
Handoff
↓
instructions
tools
handoff_functions

2. 核心概念：Agent与Handoffs

Agent（智能体）

每个Agent包含：

  • instructions：系统指令，定义Agent行为
  • tools：可调用的工具函数
  • handoff_functions：交接目标列表

Handoff（交接）

通过函数返回实现智能体之间的切换：

# 交接函数示例
def transfer_to_sales():
    """切换到销售智能体"""
    return sales_agent

# 在Agent执行中返回
result = agent.execute("我想购买产品")
if result.need_sales:
    return sales_agent  # 自动转交

3. 智能体交互流程

Swarm的交互模式非常简洁：

1
当前Agent处理用户请求
2
如需切换，返回目标Agent
3
新Agent继承上下文继续处理

4. 消息传递机制

Swarm通过函数返回值传递消息和控制权：

# Swarm 完整示例
from swarm import Swarm, Agent

client = Swarm()

def transfer_to_sales():
    return sales_agent

def transfer_to_support():
    return support_agent

sales_agent = Agent(
    name="销售",
    instructions="你是一个热情的销售代表",
    functions=[transfer_to_support]
)

support_agent = Agent(
    name="客服",
    instructions="你是一个专业的客服代表",
    functions=[transfer_to_sales]
)

triage_agent = Agent(
    name="接待",
    instructions="""你是接待员，根据用户需求转接：
    - 购买相关 → 销售
    - 问题咨询 → 客服""",
    functions=[transfer_to_sales, transfer_to_support]
)

# 启动对话
response = client.run(
    agent=triage_agent,
    messages=[{"role":"user", "content":"我想咨询产品"}]
)

5. 上下文管理

Swarm的上下文管理遵循以下原则：

  • 当前Agent的完整上下文会传递给下一个Agent
  • 通过函数参数可以携带额外的状态信息
  • 不提供持久化存储，所有状态在内存中

6. 任务规划

Swarm本身不提供显式的任务规划，规划逻辑由开发者通过Handoffs实现：

  • 接待Agent根据用户意图决定转接
  • 每个Agent决定何时转交及转交给谁
  • 复杂流程需要开发者自行设计状态机

7. 工作流编排

Swarm的工作流通过Handoff链实现：

# 复杂工作流示例
Agent → Handoff → Agent → Handoff → Agent → ...

9. 优缺点与适用场景

✅ 优点

  • 极简设计：核心概念只有两个
  • 易于理解：适合入门学习
  • 轻量快速：几乎没有额外开销
  • OpenAI官方：品质有保障

❌ 缺点

  • 功能有限：缺少高级编排能力
  • 无记忆存储：状态不持久化
  • 复杂场景受限：不适合复杂工作流
  • 实验性：已被Agents SDK取代

适用场景：轻量级路由、客服系统、多轮对话分流、简单的多智能体协作

  • 适用面窄：主要针对软件开发场景
  • 流程固定：难以适应灵活需求
  • 资源消耗大：多个LLM调用成本高
  • 学习曲线：SOP设计需要一定经验

适用场景：软件代码生成、需求到代码的完整流水线、自动化软件开发团队

1. 框架整体架构

MAR（Multi-Agent Router）是一个基于图结构的本地多智能体系统，通过学习优化的方式确定智能体之间的空间和时间连接关系。系统包含MasRouter（路由决策）和Graph（图执行引擎）两个核心组件。

MAR 架构分层

MasRouter（路由决策）
↓
Graph（图执行引擎）
↓
Agent 1 | Agent 2 | ... | Agent N
↓
Decision Node（决策节点）
Clarify Manager（澄清管理）

2. 核心智能体角色

MAR定义了基于领域的角色系统：

📐 Math Roles（数学领域）

MathSolver、Mathematician、MathAnalyst、AlgorithmEngineer等

专注数学推理、计算、证明

💻 Code Roles（代码领域）

ProgrammingExpert、BugFixer、AlgorithmDesigner等

专注代码生成、调试、优化

🧠 Commonsense Roles（常识领域）

Scientist、Economist、Historian、Critic等

专注知识推理和常识问答

🎯 Decision Node（决策节点）

整合多智能体输出，做出最终决策

支持FinalRefer等多种决策方法

3. 智能体交互流程

MAR通过空间边（Spatial）和时间边（Temporal）定义智能体间的交互：

1
MasRouter接收任务 → 分析查询内容
2
路由决策 → 选择协作方法、角色、LLM
3
构建图结构 → 根据决策构建智能体图
4
执行图 → 按拓扑顺序执行各节点
5
决策节点汇总 → 综合输出最终答案

5. 记忆与上下文共享

MAR的记忆系统包含两部分：

  • 空间上下文（Spatial Info）：当前执行轮中，依赖节点的最后输出
  • 时间上下文（Temporal Info）：历史执行轮中，节点的最后记忆
# Node.get_spatial_info 方法
def get_spatial_info(self):
    spatial_info = {}
    for predecessor in self.spatial_predecessors:
        spatial_info[predecessor.id] = {
            "role": predecessor.role,
            "output": predecessor.outputs[-1]
        }
    return spatial_info

# Node.get_temporal_info 方法
def get_temporal_info(self):
    temporal_info = {}
    for predecessor in self.temporal_predecessors:
        temporal_info[predecessor.id] = {
            "role": predecessor.role,
            "output": predecessor.last_memory['outputs'][-1]
        }
    return temporal_info

7. 工作流编排

MAR的Graph执行引擎支持动态工作流：

# Graph.run 方法
def run(self, inputs, num_rounds=2):
    for round in range(num_rounds):
        # 1. 构建空间连接（基于学习的边权重）
        log_probs += self.construct_spatial_connection()
        
        # 2. 构建时间连接
        log_probs += self.construct_temporal_connection(round)
        
        # 3. 拓扑排序执行
        in_degree = {node_id: len(node.spatial_predecessors) 
                     for node_id, node in self.nodes.items()}
        zero_in_degree_queue = [node_id for node_id, deg 
                                in in_degree.items() if deg == 0]
        
        while zero_in_degree_queue:
            # 执行节点
            current_node_id = zero_in_degree_queue.pop(0)
            self.nodes[current_node_id].execute(inputs)
            
            # 更新后继节点的入度
            for successor in self.nodes[current_node_id].spatial_successors:
                in_degree[successor.id] -= 1
                if in_degree[successor.id] == 0:
                    zero_in_degree_queue.append(successor.id)
        
        # 4. 更新记忆
        self.update_memory()
    
    # 5. 决策节点汇总
    self.connect_decision_node()
    self.decision_node.execute(inputs)
    return self.decision_node.outputs

9. 优缺点与适用场景

✅ 优点

  • 可学习的拓扑：边权重可优化，适应不同任务
  • 多LLM支持：支持不同LLM分配给不同角色
  • 空间+时间建模：同时建模当前轮和历史信息
  • 澄清机制：内置ClarifyManager进行结果校验

❌ 缺点

  • 复杂度较高：图结构和路由机制较复杂
  • 领域特定：主要针对数学和推理任务
  • 资源消耗：VAE学习和多轮执行带来开销
  • 文档较少：开源但社区支持有限

适用场景：数学推理任务（如GSM8K、MATH）、多智能体协作推理、需要优化的拓扑结构
