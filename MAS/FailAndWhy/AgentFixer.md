# AgentFixer: 从失败检测到修复建议

> [!NOTE|label:重点]
> 
> 本文综合介绍多智能体系统中"从失败中学习并自动修复"的研究方向，涵盖错误纠正图框架、自动修复建议生成等前沿工作。
>
> 核心问题：检测到失败后，如何自动生成有效的修复方案并执行？

## 一、研究动机

失败归因解决了"谁在何时犯了错"的问题，但仅仅知道错误在哪里是不够的。MAS 可靠性优化的完整链条是：

$$\text{失败检测} \rightarrow \text{归因定位} \rightarrow \text{修复建议生成} \rightarrow \text{修复执行} \rightarrow \text{验证}$$

本文关注后半段：如何从归因结果出发，自动生成和执行修复方案。

## 二、错误纠正的主要范式

### 2.1 Self-Refine 范式

最简单的纠错方式是让智能体自我反思和修正（Madaan et al., 2023）：

1. 智能体生成初始输出
2. 同一个智能体对输出进行批评（critique）
3. 根据批评修改输出
4. 重复步骤 2-3 直到满意

**局限性**：Self-Refine 依赖智能体自身的判断能力。如果智能体本身的推理能力不足以发现错误，自我修正就会失效。研究表明，LLM 的自我纠错能力在没有外部反馈的情况下非常有限。

### 2.2 层次化错误纠正图框架

A Hierarchical Error-Corrective Graph Framework（层次化错误纠正图框架）提出了更系统化的纠错方法：

**核心思想**：将 MAS 的执行过程建模为有向图，其中节点是智能体动作，边是信息流。当检测到错误时，在图上进行局部修复，而非重新执行整个流程。

**层次化结构**：
- **任务层**：将复杂任务分解为子任务图
- **智能体层**：每个子任务由一个或多个智能体负责
- **动作层**：每个智能体的执行步骤

**错误纠正流程**：
1. 在动作层检测到异常输出
2. 沿图结构向上追溯，确定受影响的子任务范围
3. 只重新执行受影响的子图，保留未受影响部分的结果
4. 将修复后的结果重新注入主图，继续执行

这种方法的优势在于：避免了全局重试的高成本，只在必要的范围内进行修复。

### 2.3 Can Agents Fix Agent Issues? (Rahardja et al., 2025)

这项工作直接探讨了一个关键问题：智能体能否修复其他智能体的问题？

**实验设计**：
- 收集多个 MAS 系统的失败案例
- 让另一个 LLM 智能体分析失败原因并提出修复建议
- 评估修复建议的质量和可执行性

**关键发现**：
- LLM 能够识别大部分失败类型（准确率 ~70%）
- 但生成的修复建议质量参差不齐：对于格式错误、指令遵循失败等简单问题，修复成功率较高；对于推理错误、幻觉等深层问题，修复成功率较低
- 修复智能体的能力高度依赖于其自身的推理能力——用更强的模型做修复效果更好

### 2.4 SHIELDA (Zhou et al., 2025)

SHIELDA 提出了结构化的异常处理框架：
- 为 MAS 工作流中的每个步骤定义预期行为和异常条件
- 当检测到异常时，触发预定义的异常处理程序
- 异常处理程序可以是：重试、回退、替换智能体、请求人工干预等

## 三、修复策略的分类

| 修复策略 | 适用场景 | 成本 | 成功率 |
|----------|----------|------|--------|
| 简单重试 | 随机性错误（如格式问题） | 低 | 中 |
| Prompt 修改 | 指令遵循失败 | 低 | 高 |
| 步骤级重执行 | 单步推理错误 | 中 | 中 |
| 智能体替换 | 能力不足 | 中 | 高 |
| 拓扑调整 | 协作失败 | 高 | 中 |
| 全局重试 | 严重错误 | 高 | 低 |

## 四、开放问题

1. **修复的可靠性**：如何保证修复操作本身不会引入新的错误？
2. **修复的效率**：如何最小化修复所需的 LLM 调用次数？
3. **修复的泛化性**：在一个任务上学到的修复策略能否迁移到其他任务？
4. **人机协作**：哪些修复应该自动执行，哪些需要人工确认？
