# 失败归因：谁在何时引发了错误

> [!NOTE|label:重点]
> 
> 本文综合介绍多智能体系统中的自动失败归因（Failure Attribution）方法，即如何自动定位"哪个智能体在哪个步骤引发了任务失败"。
>
> 这是 MAS 可靠性研究的核心问题之一，也是实现自动纠错的前提。

## 一、问题定义

给定一个多智能体系统的执行轨迹 $\tau = (s_1, a_1, s_2, a_2, ..., s_T, a_T)$，其中 $s_t$ 是第 $t$ 步的系统状态，$a_t$ 是某个智能体执行的动作。当最终任务失败时，失败归因的目标是：

1. **定位失败步骤**：找到导致失败的关键步骤 $t^*$
2. **识别责任智能体**：确定是哪个智能体 $\text{agent}(a_{t^*})$ 的动作导致了失败
3. **分析失败原因**：判断失败属于哪种类型（推理错误、幻觉、协调失败等）

## 二、代表性方法

### 2.1 WhoAndWhen (Zhang et al., 2025)

Zhang et al. 提出了自动化的失败归因框架，核心思想是将归因问题转化为因果推断问题：

**反事实分析**：对于轨迹中的每个步骤 $t$，构造一个"如果这一步做对了会怎样"的反事实场景。如果替换步骤 $t$ 的输出后任务成功，则步骤 $t$ 是失败的关键原因。

**归因得分**：定义每个步骤的归因得分为：

$$\text{attr}(t) = P(\text{success} | \text{do}(a_t = a_t^*)) - P(\text{success} | a_t)$$

其中 $a_t^*$ 是步骤 $t$ 的"正确"动作（通过 oracle 或强模型生成）。归因得分越高，说明该步骤对失败的贡献越大。

### 2.2 AgenTracer (Zhang et al., 2025)

AgenTracer 从执行轨迹中构建因果图，通过图分析定位失败源头：

1. **轨迹解析**：将执行轨迹解析为有向无环图（DAG），节点是智能体动作，边是信息依赖关系
2. **异常检测**：对每个节点的输出进行质量评估，标记异常节点
3. **根因追溯**：从最终失败节点出发，沿因果图反向追溯，找到最早的异常节点作为根因

### 2.3 TRAIL (Deshpande et al., 2025)

TRAIL 结合了轨迹推理和问题定位：
- 使用 LLM 对执行轨迹进行逐步推理分析
- 识别轨迹中的"转折点"——从正确执行转向错误执行的关键步骤
- 生成结构化的归因报告，包括失败类型、责任智能体和建议修复方案

### 2.4 RAFFLES (Zhu et al., 2025)

RAFFLES 提出了基于推理的失败归因方法：
- 将归因问题分解为多个子问题：任务是否完成？哪个子目标未达成？哪个步骤导致子目标失败？
- 使用 Chain-of-Thought 推理逐步缩小失败范围
- 支持多层次归因：从系统级到智能体级到步骤级

## 三、归因方法的评估

评估失败归因方法的关键指标：

| 指标 | 含义 |
|------|------|
| 定位准确率 | 正确识别失败步骤的比例 |
| 智能体识别准确率 | 正确识别责任智能体的比例 |
| 原因分类准确率 | 正确判断失败类型的比例 |
| 归因效率 | 完成归因所需的 LLM 调用次数 |

## 四、从归因到修复

失败归因是实现自动修复的前提。归因结果可以指导以下修复策略：

- **步骤级修复**：只重新执行失败步骤，保留正确步骤的结果
- **智能体级修复**：替换或重新配置责任智能体（如更换 prompt、更换模型）
- **系统级修复**：调整智能体间的通信拓扑或协作流程

这也是本研究方向的核心目标：构建一套从"失败检测 → 归因定位 → 自动修复"的完整流程。
