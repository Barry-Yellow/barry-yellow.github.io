为什么失效处理这么复杂？

❌ 传统程序的错误处理

  • 函数抛异常，上层捕获
  • 失败就是失败，直接返回错误
  • 用户看到错误信息，自己决定下一步
  • 状态简单：成功或失败

✓ AI Agent 的失效处理

  • 失败后模型还能继续推理
  • 会话历史必须保持 API 合法
  • 错误要变成模型可理解的上下文
  • 状态复杂：重试、降级、恢复、补偿...

核心挑战：一旦模型发出了 tool_use，如果后续没有对应的 tool_result，整段会话历史就会变成 API 非法输入，无法继续对话。

2 / 12

核心设计理念：三个不变量

1

消息轨迹必须 API 合法

不能留下孤立的 tool_use
不能让 thinking block 处在非法位置
恢复后的消息必须能被 API 接受

2

会话必须可恢复

用户消息要尽早落盘
辅助状态（compaction、todo、worktree）也要能恢复
不依赖进程现场，依赖 transcript

3

错误要变成可理解的上下文

工具失败转成 tool_result is_error=true
模型下一轮仍有机会自我纠偏
不是把主循环炸掉

"系统维护的不是函数执行成功/失败，而是这三个底层不变量"

错误分类体系

基础错误类型

AbortError

人为中断、控制流取消、子任务取消
不是普通业务失败，是"本轮流程被主动停止"

ShellError

携带 stdout、stderr、exit code、interrupted
区分：命令失败 vs 被中断 vs 内部异常

TelemetrySafeError

用户可见错误 vs 遥测可见错误分离
避免敏感信息泄露到日志

中断 vs 失败

系统不是靠字符串匹配"用户取消了"，
而是靠统一类型守卫判断"这是不是取消型错误"。

⚠️
关键区分：
中断 = 不记成严重系统错误
失败 = 需要恢复或上报

API 层：自动修复优先

核心哲学：先做自动修复，再决定是否向上报失败

withRetry 的四重职责

判断是否值得重试

不是所有错误都重试，比如 4xx 客户端错误

决定是否刷新 client/token

401、OAuth revoke、stale socket 都会触发重建

决定模型切换策略

继续原模型 / 退出 Fast Mode / 切 fallback

管理等待与心跳

长等待切成心跳块，持续 yield 状态

典型场景处理

401 / 403 认证错误

刷新 token → 重建 client → 重试
不是直接报错让用户手动修

429 / 529 限流

前台查询重试，后台任务不重试
Fast Mode 有特殊退避策略
连续 529 触发 fallback

无人值守模式

可长期等待并持续重试
避免宿主环境误判 session 死亡

Query 主循环：状态机而非单次请求

query.ts 不是单次请求函数，而是一个循环状态机，维护着 messages、toolUseContext、turnCount、transition 等状态

关键机制

1

补 missing tool_result

只要 tool_use 已发出，后续不管什么情况，都补一个 tool_result（哪怕是错误）

2

Withholding 可恢复错误

prompt too long 等错误先压住，恢复成功后再决定是否暴露

3

Model fallback 消息隔离

先封口旧轨迹，再开启新轨迹，不混用 tool_use_id

状态机维护

  • messages - 消息历史
  • toolUseContext - 工具执行上下文
  • maxOutputTokensRecoveryCount - 恢复计数
  • hasAttemptedReactiveCompact - 是否尝试过压缩
  • pendingToolUseSummary - 待处理工具
  • turnCount - 回合计数
  • transition - 状态转换
💡
不是"失败后回到函数外重新来一次"
而是在同一个回合状态机里做受控推进
7 / 12

工具层：包装失败而非抛出异常

核心理念：工具失败是本轮对话的一个"可被模型消费的事件"，不是默认要把主循环炸掉的致命异常

工具存在性检查

tool name 不存在 → 立即生成错误型 tool_result，保证闭环

参数结构校验（Zod safeParse）

缺失字段、多余字段、类型错 → 格式化成人类和模型都能读懂的说明

参数语义校验

路径不合法、值不在允许范围 → 错误型 tool_result

Hook 与权限决策

PreToolUse hook 阻止 / 权限拒绝 / classifier 风险过高 → 转成 tool_result 或 attachment

执行期失败

一般异常、AbortError、ShellError、McpAuthError → 更新状态 + PostToolUseFailure hooks + 错误型 tool_result

关键机制：tool_use/tool_result 闭环

模型发出 tool_use
→
系统记录 tool_use_id
→
执行工具
成功 → tool_result
或
失败 → tool_result is_error=true
⚠️
硬约束：只要 tool_use 已经暴露给系统，后续不管是用户中断、fallback、还是内部异常，都尽量补出一个 tool_result

这不是"更友好的提示"，而是"保证消息轨迹可继续被 API 接受"的必要条件

关键机制：上下文溢出恢复

Prompt Too Long 恢复

检测 withholding 的 PTL

错误先压住，不立即暴露

尝试 context collapse

轻量恢复，保留原有上下文颗粒度

尝试 reactive compact

更重，把上下文重写成压缩摘要

仍失败才暴露错误

给用户明确的上下文溢出提示

Max Output Tokens 恢复

提升输出上限

如果还没做过更高上限尝试

插入续写指令

meta user message：直接继续，不要道歉，不要重复 recap

最多 3 次恢复

仍失败才真正报错

💡
语义层面续写
不是传统断点恢复，而是让模型在语义层面继续原任务

流式并发工具执行的失败传播

StreamingToolExecutor 解决的问题：工具一边 streaming 一边执行，多个工具并发，其中一个失败后其他怎么办？

状态管理

  • queued - 排队等待
  • executing - 正在执行
  • completed - 已完成
  • yielded - 已输出结果

并发安全工具

可以并行执行，提高效率

非并发安全工具

必须串行执行，保证正确性

失败传播策略

Sibling Error

一个工具失败，为其他并发工具生成 synthetic error result

Streaming Fallback

模型切换时，直接 discard 当前批次执行器

⚠️
关键：失败传播不是"炸掉所有"，而是让每个工具都有明确的结束状态

设计原则总结

🎯

错误分层处理

API 层、回合层、工具层、权限层、会话恢复层、多智能体任务层各司其职

🔗

保持链路连续

tool_use 必有 tool_result，消息轨迹始终 API 合法

🔄

先恢复再暴露

可恢复错误先压住，尝试恢复成功后再决定是否报错

💾

依赖 transcript

不依赖进程现场，从消息日志重建可继续推理的会话状态

🧹

恢复前清洗历史

去掉不完整 tool_use、孤立 thinking、坏掉的 tool_result

💬

错误变成上下文

失败包装成模型可理解的消息，让模型有机会自我纠偏

"失败大多数不会立即终止整个会话，而是变成消息或任务状态，让主对话还能继续"
