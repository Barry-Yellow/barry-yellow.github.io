1问题定义与动机

传统方法的问题

发现错误后，从错误点开始全量重跑所有后续步骤

  • 假设"错误点之后的所有内容都失效"
  • 大量无意义重跑，浪费计算资源
  • 无法区分"真正被污染"和"只是时间上在后面"
VS

AG-Repair 的思路

将失败轨迹视为可分析、可切分、可局部修补的结构化过程

  • 通过依赖建模找出真正的信息传播路径
  • 只修复必要的最小子图
  • 保留未被污染的有效执行结果
修复目标的四个约束：
正确性 修复后轨迹满足任务目标或推进里程碑
最小干预 尽量减少需要重执行的节点数
最大复用 尽量保留原轨迹中未被污染的部分
一致性 新旧轨迹在接口、逻辑和状态上不冲突

执行轨迹的形式化定义

T = ⟨ e₁, e₂, ..., eₙ ⟩

每个事件 eᵢ 是一次可观测执行单元：agent 推理输出、工具调用、工具返回、artifact 写入、checkpoint、verifier 信号

2因果图构建 — 从线性日志到 MAICG 图结构

核心前提

多智能体执行过程不是单纯的线性链，而是具有复杂依赖关系的交互系统。一个节点是否受错误影响，不能只看它是否出现在错误点之后，还必须看它是否真正使用了错误信息。

G = (V, E) — 多智能体交互因果图 (MAICG)

agent 的一次推理输出

📋

工具执行返回结果
📦

共享对象（代码、搜索结果等）
💾

边类型 (E) — 六类因果依赖

边类型含义示例传播权重

关键区分：将"真实信息传播"和"只是时间上发生在后面"分开，这是精确边界识别的基础。

3根因定位 — 找最早、最关键、最值得修的节点

定位目标

不是回答"哪一步有错误"，而是回答：
哪一个节点最早引入了对后续任务失败真正有决定性影响的偏差，并且最值得被修复。

定位的不是"最显眼的错误点"，而是"修复价值最高的根因点"。

Step 1：候选错误发现

硬信号（优先）

  • 显式异常 / 工具调用失败
  • 协议违规（如声称搜索却无真实证据）
  • verifier 失败 / 约束断言失败

软信号（补充）

  • 结果偏离任务目标
  • 下游输出与上游事实冲突
  • 边缘条件未覆盖 / 上下文信息丢失

所有信号统一记录在节点的 signals 字段中

Step 2：根因评分公式

Sroot(v) = α·A(v) + β·I(v) + γ·D(v) + δ·R(v)
分量含义说明
A(v)异常强度该节点上观测到的错误证据强弱
I(v)影响强度从该节点向后能到达多少潜在受影响节点
D(v)偏离程度节点结果与任务目标/约束之间的偏差
R(v)修复收益修复该节点的潜在恢复价值

默认策略：硬信号优先 → A(v) 和 I(v) 权重更高

Step 3：根因排序与选择策略

1
按 root_score 排序
保留前若干个高价值根因点
2
首选单根因修复
缩小初始范围，便于边界传播
3
失败后升级
首次修复失败 → 多根因联合修复

5.1

4边界识别 — 区分 Clean / Partial / Invalid

为什么必须识别边界？

如果只知道"第 50 步错了"，却不知道后续哪些节点真正被污染，就无法做到最小干预。传统方法将错误点之后全部步骤判为失效，带来大量无意义重跑。AG-Repair 把错误传播范围建模为图上的可计算问题。

污染传播模型

P(v) = maxu∈Pred(v) P(u) · w(u,v) · λ
P(v)节点 v 的污染分数
Pred(v)所有指向 v 的前驱节点
w(u,v)边权重（按边类型不同）
λ = 0.85衰减系数
直观原则：显式数据依赖 (1.0) 和工具结果依赖 (0.95) 最容易传播错误；纯时序关系 (0.1) 只产生弱影响

复用评分公式

污染分数不能直接决定是否重跑 — 部分污染的节点可能仍保留大量可用结构

Sreuse = 0.5·Scorrect + 0.3·(1−P) + 0.2·Scompat
Scorrect节点本地正确性（权重 50%）
1 − P未污染程度（权重 30%）
Scompat与修复后上下文的兼容性（权重 20%）

最终缩放到 0–100 范围

节点三级分类 → 直接决定修复动作

Clean
Sreuse ≥ 80
完全可复用 → 保留
Partial
60 ≤ Sreuse < 80
部分可复用 → 优先局部修补
Invalid
Sreuse < 60
不可复用 → 重放或重写
分支级升级策略：若某分支 Invalid 节点占比 > 40%，或包含高层 planning node 错误 → 整体升级为 REWRITE_BRANCH，避免在大面积崩坏的分支上做低收益小修小补。

6增量重执行 — 只重跑必要子图，不动安全区域

核心思想

关键不是"重新运行模型"，而是"只重新运行必要部分，同时让保留部分继续有效"。

能力 1
从合适的 checkpoint 恢复状态
能力 2
对无需修改的旧结果做 journal replay
能力 3
对重跑节点注入修复后的最小依赖上下文

恢复起点选择

从最靠近根因节点、且位于其之前的 checkpoint 恢复

  • 尽量缩短重放路径
  • 避免将已受污染的状态带入恢复流程

Active Dependency Frontier

不将整条原始轨迹作为上下文，只注入依赖前沿：

  • 直接依赖的上游节点
  • 相关的共享 artifact
  • 修复后的最新中间结果
  • 当前分支必要的规划信息

显著减少长上下文干扰，降低随机性

各动作的具体执行方式

动作执行方式影响范围
PRESERVE完全保留，不重新执行零开销
PATCH对原节点内容局部更新（替换错误参数、修补结论、注入缺失字段）单节点内部
REPLAY_TOOL仅重新执行该工具调用，其他节点继续复用单工具调用
RERUN_NODE重新运行当前节点，使用修复后的依赖前沿作为输入单节点
RERUN_BRANCH对整个子分支重新运行，保留其它分支与公共节点单分支
REWRITE_BRANCH对整段分支重新生成（局部修补已不经济时）单分支（完全重写）
局部失败的自动升级机制：修复后未通过局部一致性校验 → 自动升级一级

5修复动作 — 分类·选择·增量重执行示意

8.1

7一致性验证与完整流程示例

局部一致性检查

  • 修补后的输出是否满足下游输入接口
  • 被引用的 artifact 是否仍存在
  • 工具结果与调用状态是否匹配
  • 修复后节点与保留节点之间是否产生直接矛盾

局部一致性失败 → 当前动作粒度过小 → 升级修复动作

全局一致性检查

  • 是否满足任务成功标准
  • 任务里程碑是否前进 (milestone_forward)
  • 是否引入了新的错误
  • 最终结果是否与原目标兼容

即使未完全成功，只要显著缩小失败范围或推动关键里程碑，也视为有效改进

完整流程示例 — 网页搜索与事实整合任务

Step 1
采集轨迹：planner 规划 → researcher 调用搜索 → 获得结果 → writer 写摘要
Step 2
注入错误：摘要节点被替换为"伪造搜索证据后直接总结"，标记协议违规 + 验证失败
Step 3
构建 MAICG：4 个节点 + artifact 节点，建立 plan/data/tool/artifact/context 边
Step 4
根因定位：末端摘要节点检测到 protocol_violation + verification_failure + deviation_score=0.8 → 高根因分数
Step 5
边界传播：末端错误 → 污染范围集中在最后节点，前面的 planner/tool_call/tool_result 判为可复用
Step 6
修复动作：前面步骤 PRESERVE，末端节点 RERUN_NODE（证据伪造 → 升级为真实重放）
Step 7
增量重放：从最近 checkpoint 恢复，保留已有搜索结果，仅对末端摘要节点重新执行
Step 8
一致性验证：检查与旧搜索结果一致性 + 无伪造内容 + 里程碑前进 → 输出 repaired run

∑方法总结 — 三层核心创新

1. 错误边界识别

把错误传播范围从经验判断变成图上的量化问题 — 通过 MAICG 上的污染传播模型 + 复用评分，精确区分 Clean / Partial / Invalid

2. 修复建议生成

把开放式"怎么修"变成受限动作空间中的成本感知决策 — 六种动作按代价梯度排列，层层升级，可解释可审计

3. 增量重执行

把"从错误点之后全部重来"变成"从最近安全状态出发，仅重放必要子图" — 依赖 checkpoint + journal replay + Active Dependency Frontier

AG-Repair 不只是一个 failure attribution 框架

而是一套真正面向 failure recovery 的完整方法论
