AG-Repair 方法论

多智能体系统长轨迹失效的自动化修复与增量恢复框架

轨迹采集→ 因果图构建→ 根因定位→ 影响边界识别→ 修复动作生成→ 增量重执行→ 一致性验证

1. 方法概述

AG-Repair 的目标是解决多智能体系统在长轨迹执行过程中出现的失效问题，并在尽可能保留原有有效执行结果的前提下，完成自动化修复与增量恢复。

❌ 传统方法

发现错误后从错误点开始全量重跑，假设"错误点之后的所有内容都失效"

✅ AG-Repair

将失败执行视为可分析、可切分、可局部修补的结构化过程，通过依赖建模与增量重放，将修复范围缩小到最小子图

整个框架围绕一条完整闭环展开，该闭环的设计目标有四个：

🎯 精准定位
在长轨迹中找出真正需要修复的关键错误点，而不是只做表面错误检测
📊 量化影响
对错误影响范围进行量化，区分哪些节点仍然可复用，哪些节点必须修补或重跑
💰 成本感知
用成本感知的方式生成修复方案，优先选择代价最小、风险最低的动作
🔄 增量恢复
修复后只重放必要部分，并通过一致性校验确保新轨迹正确接回原任务流程

为了实现这四个目标，AG-Repair 将多智能体运行过程统一建模为一个多智能体交互因果图，并在其上执行后续分析与恢复。

2. 问题定义

2.1 多智能体执行轨迹

设一个多智能体系统执行某个复杂任务后，产生一条执行轨迹：

$$T = \langle e_1, e_2, \ldots, e_n \rangle$$

其中每个事件 $e_i$ 表示一次可观测执行单元，例如：

事件类型说明
agent_step某个 agent 输出一步推理结果
tool_call某个 agent 发起一次工具调用
tool_result某个工具返回一次执行结果
artifact某个共享 artifact 被写入或更新
checkpoint系统生成一次 checkpoint
verifier_signalverifier 发出一次失败信号
关键转变

在传统方法中，这条轨迹往往只被看作一段时序日志；而在 AG-Repair 中，我们将其转换为图结构，以保留事件之间的真实依赖关系。

2.2 修复目标

给定一条失败轨迹 $T$，修复系统需要构造一条新轨迹 $T'$，满足以下条件：

✅ 正确性
修复后的轨迹能够满足任务目标或显著推进任务里程碑
✂️ 最小干预
尽量减少需要重执行的节点数
♻️ 最大复用
尽量保留原轨迹中未被污染的部分
🔗 一致性
新轨迹与保留轨迹之间在接口、逻辑和状态上不发生冲突
核心优化目标

AG-Repair 优化的不是单一"成功率"，而是在"修复成功"和"重跑代价"之间寻找平衡。

3. 统一表示：多智能体交互因果图

3.1 从线性轨迹到图结构

AG-Repair 的核心前提是：多智能体执行过程不是单纯的线性链，而是一个具有复杂依赖关系的交互系统。一个节点是否受错误影响，不能只看它是否出现在错误点之后，还必须看它是否真正使用了错误信息。

因此，我们构建一个多智能体交互因果图（Multi-Agent Interaction Causal Graph, MAICG）：

$$G = (V, E)$$

其中 $V$ 表示轨迹中的结构化节点集合，$E$ 表示节点之间的依赖边集合。

3.2 节点类型

节点类型说明标签
agent_step某个 agent 的一次输出或推理步骤Agent
tool_call一次工具调用请求Tool
tool_result一次工具执行返回Tool
artifact共享对象（代码文件、搜索结果等）Artifact
checkpoint可恢复状态快照State
verifier_signal验证器发出的失败或约束违例信号Signal
设计意义

1. 使系统不再只能分析"agent 说了什么"，而是同时追踪工具调用、共享对象和恢复状态。

2. 为后续增量重放提供了更细粒度的修复单元。

3.3 边类型

边表示节点之间的因果依赖。AG-Repair 使用六类边：

边类型含义传播权重
data显式数据依赖，如某一步直接读取前一步输出1.0
tool工具调用与工具结果之间的对应关系0.95
artifact节点对共享 artifact 的读写依赖0.85
plan规划节点与其子任务执行节点之间的依赖0.7
context同一 agent 跨轮对话的上下文延续关系0.5
temporal纯时序相邻关系，仅表示顺序0.1
核心区分

这种类型化依赖使系统能够将"真实信息传播"和"只是时间上发生在后面"区分开来。

3.4 图构建过程

图构建分为两个阶段：

1
阶段一：节点实例化
从原始轨迹中提取标准化事件 → 映射为统一的 TraceEvent → 转化为 TrajectoryNode。如果事件写入了 artifact，则额外创建 artifact 节点并显式连接写入边。
2
阶段二：依赖补全
在节点存在之后，补充不同类型的依赖边：context 边、tool 边、data 边、artifact 边、temporal 边。最终得到的图既保留了执行顺序，也保留了真正的信息流动结构。

4. 轨迹采集与可恢复状态管理

4.1 采集目标

AG-Repair 的采集层不只是记录日志，而是同时完成三件事：

📝 结构化事件流
记录完整的执行事件序列，保留原始轨迹信息
📸 可恢复状态快照
在关键节点捕获 checkpoint，支持事后局部恢复
🔧 可回放工具日志
记录工具调用的完整输入输出，支持旧结果回放

4.2 事件存储

📄 JSONL 顺序落盘

适合做回放与审计，保留原始轨迹

🗄️ SQLite 索引

适合做快速定位与分析，索引关键字段

SQLite 索引的关键字段包括：

字段说明
run_id运行标识
step_id步骤标识
agent_idAgent 标识
tool_call_id工具调用标识
checkpoint_idCheckpoint 标识

4.3 Checkpoint 设计

Checkpoint 是 AG-Repair 增量恢复的基础。系统在以下关键时刻自动捕获：

1
规划器输出新子任务后
2
Agent 即将调用工具前
3
工具返回后
4
Artifact 被写入后
5
Verifier 发出失败信号后
6
某一执行分支结束时

每个 checkpoint 至少包含：

内容说明
对话上下文摘要当前对话状态
Agent 内部状态摘要Agent 运行时状态
已确认 artifact 索引已写入的共享对象
依赖祖先节点最近若干真实依赖
工具调用 journal 引用可回放日志
环境元信息与随机种子环境可复现信息

4.4 Tool Journal

对于工具调用，AG-Repair 记录一个可回放日志。当一个工具节点未被纳入修复范围时，系统在恢复时优先回放旧结果而不是重新调用外部环境。

双重目的

1. 减少不必要的代价

2. 避免因外部环境变化导致恢复结果与原上下文不一致

因此，checkpoint 和 journal 一起构成了"可恢复执行底座"。

5. 根因定位

5.1 根因定位的目标

核心问题

AG-Repair 不是简单回答"哪一步有错误"，而是要回答：哪一个节点最早引入了对后续任务失败真正有决定性影响的偏差，并且最值得被修复。

这意味着系统定位的不是"最显眼的错误点"，而是"修复价值最高的根因点"。

5.2 候选错误发现

根因定位首先从所有节点中筛选候选错误点。系统优先使用可观测的硬信号，然后再补充软信号：

🔴 硬信号（优先）
  • 显式异常
  • 工具调用失败
  • 协议违规
  • Verifier 失败
  • 约束断言失败
🟡 软信号（补充）
  • 结果偏离任务目标
  • 下游输出与上游事实冲突
  • 边缘条件未覆盖
  • 上下文信息丢失

5.3 根因评分

对每个候选节点 $v$，系统计算一个根因分数：

$$S_{root}(v) = \alpha \cdot A(v) + \beta \cdot I(v) + \gamma \cdot D(v) + \delta \cdot R(v)$$
符号含义说明
$A(v)$异常强度该节点上观测到的错误证据强弱
$I(v)$影响强度从该节点向后能到达多少潜在受影响节点
$D(v)$偏离程度该节点结果与任务目标或约束之间的偏差
$R(v)$修复收益修复该节点的潜在恢复价值
策略

系统默认采用"硬信号优先、语义信号补充"的策略，因此异常强度与影响强度具有更高权重。

5.4 根因排序结果

系统将所有候选节点按 root_score 排序，并保留前若干个高价值根因点。对于第一版系统，通常只选择排名最高的节点作为主要修复锚点。这样做的原因：

  1. 有利于缩小初始修复范围
  2. 便于后续做边界传播
  3. 可以通过增量修复结果反向检验该根因是否正确

如果第一次修复失败，系统才会考虑升级为多根因联合修复。

6. 错误影响边界识别

6.1 为什么必须识别边界

如果只知道"第 50 步错了"，却不知道后续哪些节点真正被污染，就无法做到最小干预。传统方法往往将第 50 步之后的全部步骤判为失效，这会带来大量无意义重跑。

关键创新

AG-Repair 把错误传播范围建模为一个可计算问题，而不是简单地"一刀切"。

6.2 污染传播模型

设根因节点为 $r$，系统在图上对其进行前向传播。传播采用衰减模型：

$$P(v) = \max_{u \in Pred(v)} P(u) \cdot w(u,v) \cdot \lambda$$
符号含义
$P(v)$节点 $v$ 的污染分数
$Pred(v)$所有指向 $v$ 的前驱节点
$w(u,v)$边权重（影响强度）
$\lambda$衰减系数，默认 0.85

默认边权重设置：

边类型权重直观含义
data1.0显式数据依赖，最容易传播错误
tool0.95工具结果依赖，几乎等同数据
artifact0.85共享对象依赖
plan0.7规划依赖，中等传播
context0.5上下文延续，较弱传播
temporal0.1纯时序关系，极弱传播

6.3 节点复用评分

污染分数本身并不能直接决定是否重跑，因为一个节点即使受到了部分污染，也可能仍然保留大量可用结构。因此，AG-Repair 进一步定义一个复用评分：

$$S_{reuse} = 0.5 \cdot S_{correct} + 0.3 \cdot (1 - P) + 0.2 \cdot S_{compat}$$
符号含义权重
$S_{correct}$节点本地正确性0.5
$1 - P$污染分数的反向（越干净越高）0.3
$S_{compat}$节点与修复后上下文的兼容性0.2

该分数最终被缩放到 0 到 100 的范围内。

6.4 节点分类

根据复用评分，节点被分为三类：

类别条件后续动作标签
完全可复用（Clean） $S_{reuse} \ge 80$ 保留 Clean
部分可复用（Partial） $60 \le S_{reuse} < 80$ 优先局部修补 Partial
不可复用（Invalid） $S_{reuse} < 60$ 纳入重放或重写范围 Invalid

6.5 分支级升级策略

如果某一分支中 Invalid 节点占比超过 40%，或者该分支包含高层 planning node 的错误，则系统将该分支整体升级为 REWRITE_BRANCH。

升级原则

避免在已经大面积崩坏的分支上继续做低收益的小修小补。

7. 修复动作生成

7.1 目标

AG-Repair 并不让模型自由生成"应该怎么修"，而是将修复规划视为一个受限决策问题。系统首先识别错误模式，再在固定动作空间中做成本感知选择。

7.2 错误模式

第一版系统支持五类主要错误模式：

evidence_fabrication
事实获取失败 / 证据伪造
tool_call_error
工具参数错误 / 调用失败
context_drop
上下文传递失败 / 信息丢失
reasoning_bug
局部逻辑错误 / 推理偏差
plan_bug
高层规划错误 / 任务分解失败

7.3 动作空间

系统支持六种修复动作，具有明确的代价梯度：

PRESERVE
→
PATCH
→
REPLAY_TOOL
→
RERUN_NODE
→
RERUN_BRANCH
→
REWRITE_BRANCH

代价从低到高 →

动作说明代价
PRESERVE完全保留最低
PATCH原位局部修补低
REPLAY_TOOL仅重放工具调用中低
RERUN_NODE仅重跑单节点中
RERUN_BRANCH重跑分支高
REWRITE_BRANCH重写分支最高

7.4 决策逻辑

在默认策略下：

节点状态默认动作
Clean 节点PRESERVE
Partial 节点优先 PATCH
Invalid 的工具节点优先 REPLAY_TOOL
Invalid 的普通执行节点优先 RERUN_NODE
规划错误优先 RERUN_BRANCH
分支级污染严重升级为 REWRITE_BRANCH
设计作用

1. 把重跑范围控制在最小必要集

2. 让修复动作可解释、可审计，而不是完全依赖一次不可控的生成结果

7.5 RepairPlan

最终，规划器输出一个 RepairPlan，其中包含：

  • 需要恢复的 checkpoint
  • 根因节点列表
  • 所有动作列表
  • 每个动作的优先级与原因
  • 修复过程中的默认假设

RepairPlan 是连接"分析阶段"和"执行阶段"的核心接口。

8. 增量重执行

8.1 核心思想

核心理念

增量重执行的关键不是"重新运行模型"，而是"只重新运行必要部分，同时让保留部分继续有效"。

这要求系统在执行时具备三种能力：

📸 Checkpoint 恢复
从合适的 checkpoint 恢复状态
📖 Journal Replay
对无需修改的旧结果进行回放
💉 依赖注入
对需要重跑的节点注入修复后的最小依赖上下文

8.2 恢复起点选择

系统默认从最靠近根因节点、且位于其之前的 checkpoint 恢复。这样做有两个优点：

  1. 尽量缩短重放路径
  2. 避免将已经受污染的状态带入恢复流程

8.3 Active Dependency Frontier

在恢复某个待重跑节点时，AG-Repair 不会将整条原始轨迹重新作为上下文输入，而是只注入该节点当前真正需要的"依赖前沿"：

  • 它直接依赖的上游节点
  • 与之相关的共享 artifact
  • 修复后的最新中间结果
  • 当前分支必要的规划信息
效果

这种输入裁剪机制可以显著减少长上下文干扰，并降低随机性。

8.4 不同动作的执行方式

动作执行方式
PRESERVE节点完全保留，不重新执行
PATCH只对原节点内容进行局部更新（替换错误参数、修补错误结论、注入缺失字段）
REPLAY_TOOL仅重新执行该工具调用，其他不受影响节点继续复用
RERUN_NODE只重新运行当前节点，并使用修复后的依赖前沿作为输入
RERUN_BRANCH对整个子分支重新运行，但仍保留其它分支与公共节点
REWRITE_BRANCH当局部修补已不经济时，对整段分支进行重新生成

8.5 局部失败的升级机制

如果某个节点修复后未通过局部一致性校验，系统会自动将其动作升级一级：

PATCH
→
REPLAY_TOOL
→
RERUN_NODE
→
RERUN_BRANCH

最多允许两轮升级，以避免陷入无限重试。

9. 一致性验证

9.1 为什么要做一致性验证

即使某个节点被成功修补，也不代表整条轨迹已经正确。局部替换后可能会引入新的接口冲突、逻辑冲突或状态漂移。因此，AG-Repair 在修复完成后必须执行验证。

9.2 局部一致性

局部一致性主要检查：

检查项说明
接口兼容修补后的输出是否满足下游输入接口
Artifact 存在性被引用的 artifact 是否仍存在
工具结果匹配工具结果与调用状态是否匹配
矛盾检测修复后节点与保留节点之间是否产生直接矛盾
升级触发

如果局部一致性失败，说明当前动作粒度过小，系统应升级修复动作。

9.3 全局一致性

检查项说明
任务成功标准是否满足任务成功标准
里程碑前进任务里程碑是否前进
新错误引入是否引入了新的错误
目标兼容最终结果是否与原目标兼容

系统输出一个 VerificationReport，包含：

  • local_consistency
  • global_success
  • milestone_forward
  • new_error_rate
  • 若干聚合指标

9.4 Milestone Forward

关键指标

AG-Repair 特别关注修复是否"推动任务向前"，而不仅仅是"减少错误"。即使一次修复尚未完全成功，只要它显著缩小了失败范围，或推动了任务关键里程碑，也应被视为有效改进。

10. 合成数据与故障注入

10.1 为什么使用合成数据

在多智能体修复系统的早期阶段，真实失败日志往往数量有限、标注成本高、复现实验困难。因此，AG-Repair 的第一版采用"合成数据优先"的策略，通过可控错误注入生成训练与验证样本。

10.2 任务类型

web_research

10.3 故障注入方式

针对每类任务，系统支持以下错误注入器：

注入器说明
evidence_fabrication证据伪造
tool_call_error工具调用错误
context_drop上下文丢失
reasoning_bug推理错误
plan_bug规划错误

每次注入都只破坏轨迹中的某一关键节点，从而能够精确标注：

  • 根因步骤
  • 污染节点集合
  • 安全节点集合
  • 建议修复动作
  • 成功修复作用域

11. 全流程执行示例

下面以一个"网页搜索与事实整合任务"为例说明 AG-Repair 的完整工作流：

1
采集轨迹
系统运行一个 web research 任务，得到核心节点：planner 规划搜索流程 → researcher 调用搜索工具 → researcher 获得搜索结果 → writer 根据搜索结果写摘要
2
注入错误
在 fault injection 阶段，将最后的摘要节点替换为"伪造搜索证据后直接总结"的版本，并标记其存在协议违规和验证失败
3
构建图
将 4 个节点和相关 artifact 节点构造成 MAICG：planner → tool call 的 plan/data 边，tool call → tool result 的 tool 边，tool result → summary 的 data/artifact 边，同 agent 连续消息的 context 边
4
根因定位
localizer 检测到最后摘要节点存在：protocol_violation = true、verification_failure = true、deviation_score = 0.8，获得较高根因分数
5
边界传播
boundary scorer 从根因节点向后传播污染。由于是末端错误节点，污染范围集中在最后节点本身，前面的 planner、tool call 和 tool result 节点大概率仍被判为可复用
6
修复动作生成
前面步骤 PRESERVE，末端错误节点 RERUN_NODE 或 PATCH。由于涉及证据伪造，系统更倾向升级为真实重放
7
增量重放
从最近 checkpoint 恢复，保留已有搜索结果，仅对末端摘要节点重新执行，并向其注入真实搜索结果与修复后的上下文
8
一致性验证
检查新摘要是否与旧搜索结果一致、是否满足"有真实证据、无伪造内容"的成功标准、是否使任务里程碑前进。若通过，则输出 repaired run
核心结论

AG-Repair 修复的不是整个轨迹，而是只对真正受污染的最小区域进行恢复。
