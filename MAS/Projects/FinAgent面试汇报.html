设计要点

GlobalState 是 LangGraph 图中所有节点共享的唯一下文对象，每个 Agent 节点只修改自己负责的字段，通过 StateGraph.add_node / add_edge / add_conditional_edges 声明式编排流转逻辑。状态不可变性由 TypedDict 类型约束保证。

条件路由机制

对话 Agent 识别意图后，通过 add_conditional_edges 将流程路由到对应节点：

路由决策函数

根据 state["intent"] 值返回目标节点名：crawl_data → 采集 Agent；summarize → 摘要 Agent；analyze → 分析 Agent；chat → 直接回复；multi → 串行执行多意图

易错点

多意图场景下需串行执行而非并行，因为后续意图可能依赖前序意图的输出（如先采集再分析），顺序错误会导致下游缺少数据。

意图识别 Prompt 设计

结构化输出约束

Prompt 中明确定义 JSON Schema：要求 LLM 输出包含 intent（枚举值）和 intent_detail（dict，含具体参数如股票代码、分析类型等），通过 few-shot 示例引导格式一致性。

易错点：LLM 输出格式不稳定

LLM 可能输出非标准 JSON（如添加注释、多余逗号、Markdown 代码块包裹）。解决方案：输出后置处理——strip() 去除首尾空白，正则提取 JSON 块，json.loads() 严格解析，失败时 fallback 到关键词匹配。

多意图拆分策略

难点：复合意图的识别与拆分

用户输入可能包含多个意图（如"爬取数据并分析"），需要在 Prompt 中引导 LLM 识别并列出所有子意图，按依赖关系排序后串行执行。

解决方案

intent_detail 中增加 sub_intents 列表字段，每个子意图独立包含 type + params。执行引擎按列表顺序串行调度，前序输出自动注入后续 context。若某子意图失败，记录错误但继续执行后续意图（部分降级策略）。

Prompt Engineering 技术深度

渐进式分析链（Progressive Analysis Chain）

分析 Agent 的 9 个技能采用模板方法模式，每个技能内部由 2-4 个渐进式 Prompt 串联执行，前一步输出作为后一步输入：

S1

数据理解

理解原始数据结构、字段含义、数据质量

S2

指标计算

根据分析类型计算关键财务指标

S3

深度分析

结合行业背景进行多维度解读

S4

结论输出

结构化输出分析结论与建议

模板方法模式

所有分析技能继承自 BaseSkill 抽象基类，统一执行框架：

BaseSkill 执行流程

validate_input() → prepare_prompt() → call_llm() → parse_output() → format_result()。子类只需重写 prepare_prompt() 和 parse_output()，保证输出格式一致性和错误处理的统一性。

易错点：Prompt 链中间输出漂移

多步 Prompt 串联时，中间步骤的 LLM 输出可能偏离预期格式，导致后续步骤解析失败。解决方案：每步输出后增加格式校验，不合法时重试（最多 2 次），仍失败则降级返回部分结果。

Prompt 工程关键技巧

1. 结构化输出约束

所有 Prompt 末尾附加 JSON Schema 定义，要求 LLM 严格按格式输出，减少后处理复杂度。

2. Few-shot 示例引导

每个技能 Prompt 包含 1-2 个完整示例，覆盖典型输入输出，显著提升格式一致性和分析质量。

3. 角色设定与领域约束

System Prompt 设定"专业金融分析师"角色，约束分析范围在金融领域，避免幻觉和无关输出。

4. 上下文窗口管理

长对话场景下需裁剪历史消息，保留最近 N 轮 + 关键摘要，防止超出 Token 限制导致截断。

采集难点与易错点

难点 1：动态渲染页面

金融网站大量使用 JS 动态渲染，传统 requests 无法获取数据。必须使用 Playwright 等浏览器自动化工具，等待 AJAX 请求完成后再提取内容。需智能等待策略（networkidle / domcontentloaded）而非固定延时。

难点 2：反爬对抗升级

网站持续更新检测手段（如 TLS 指纹、Canvas 指纹），需持续维护 stealth 插件和反检测策略。playwright-stealth 并非万能，部分站点仍需自定义 JS 注入。

易错点：超时与资源泄漏

Playwright browser/page 对象必须确保在 finally 块中关闭，否则长期运行会导致内存泄漏和僵尸进程。使用 async with 上下文管理器或 try/finally 保证资源释放。

解决方案

封装 BrowserManager 单例，统一管理浏览器生命周期；所有页面操作设置合理超时（默认 30s）；异常时自动截图保存现场，便于排查。

RRF 融合算法

Reciprocal Rank Fusion 原理

对两路检索结果分别按排名计算得分：score(d) = Σ 1/(k + rank_i(d))，k 为平滑常数（默认 60）。按融合得分降序排列取 Top-K，兼顾语义相关性和关键词精确匹配。

易错点：Embedding 模型选择

金融领域术语密集，通用 Embedding 模型可能无法准确捕获专业语义。需选择支持中文的金融领域微调模型，或在索引时对专业术语增加同义词扩展。

三层记忆管理 难点 技术深度

记忆分层架构

短期记忆

存储：内存 dict

内容：当前会话消息历史

淘汰：TTL 过期自动清除

特点：读写最快，容量有限

长期记忆

存储：本地 JSON 文件

内容：跨会话用户偏好、历史摘要

淘汰：FIFO + 容量上限

特点：持久化，跨会话保持

RAG 记忆

存储：向量数据库

内容：文档知识库、研报索引

淘汰：无淘汰，持续积累

特点：语义检索，知识增强

记忆读写流程

用户输入 → 短期记忆写入

每条消息追加到内存 dict，设置 TTL 时间戳

检索时 → 三层并行查询

短期记忆直接匹配 → 长期记忆文件读取 → RAG 向量检索

结果融合 → 注入 Prompt 上下文

按相关性排序，裁剪至 Token 预算内，注入 System Prompt

会话结束 → 长期记忆更新

提取关键信息写入 JSON，超出容量时 FIFO 淘汰最旧记录

记忆管理难点与易错点

难点：记忆一致性

三层记忆可能存在信息冲突（如短期记忆中的最新数据 vs 长期记忆中的历史数据）。解决方案：设置优先级——短期 > 长期 > RAG，高优先级覆盖低优先级。

易错点：TTL 与 FIFO 的边界条件

短期记忆 TTL 过期但会话仍在进行时，需判断是否延长 TTL 而非直接清除。长期记忆 FIFO 淘汰时需确保不丢弃高频访问的关键信息。解决方案：引入访问计数，LFU + FIFO 混合淘汰策略。

易错点：向量数据库索引更新

新文档入库后需重建或增量更新向量索引，若忘记更新会导致 RAG 检索不到最新知识。解决方案：写入后立即触发增量索引，并验证检索可命中。

失败处理与容错机制 核心重点 难点

多层容错体系

L1 — 自动重试

LLM 调用失败、网络超时等瞬态错误，指数退避重试（最多 3 次），间隔 1s → 2s → 4s

L2 — 降级策略

重试仍失败时切换备选方案：LLM Provider 切换（OpenAI → 本地模型）、采集降级（Playwright → requests）、分析降级（详细分析 → 简要概述）

L3 — 状态回滚

Agent 执行异常时，GlobalState 中 error 字段记录异常信息，已修改字段回滚到执行前快照，防止脏状态污染后续流程

L4 — 用户兜底

所有容错手段失效后，向用户返回友好的错误提示和建议操作，而非裸露的异常堆栈

容错易错点

易错点：重试风暴

多 Agent 并发重试可能导致 API 限流加剧。解决方案：全局重试协调器，限制同一时刻的重试并发数，避免雪崩效应。

易错点：状态回滚不完整

Agent 修改了多个 State 字段但中途失败时，若只回滚部分字段会导致状态不一致。解决方案：执行前深拷贝 State 快照，失败时整体回滚。

难点：部分降级的用户体验

多意图场景下部分成功部分失败时，需向用户明确说明哪些完成、哪些失败，而非笼统报错。解决方案：结果中增加 status 字段，逐项标注执行状态。

权限控制与安全 难点

域名白名单

采集 Agent 限制只访问预定义的金融站点域名，防止 SSRF 攻击和非法数据采集：

实现机制

URL 解析阶段提取域名，与白名单集合进行匹配。非白名单域名直接拒绝并返回错误提示。白名单通过配置文件管理，支持动态更新。

易错点：URL 绕过

攻击者可能通过 URL 重定向、IP 地址直连、子域名枚举等方式绕过白名单。解决方案：跟随重定向后再次校验最终 URL 域名；禁止 IP 直连；严格子域名匹配。

API Key 安全

LLM API Key 等敏感凭证的安全管理：

安全实践

所有 API Key 通过环境变量注入，绝不硬编码在源码中。.env 文件加入 .gitignore，生产环境使用密钥管理服务。日志中自动脱敏 Key 值。

LLM 输出过滤

LLM 生成内容经过后处理过滤，防止泄露系统 Prompt、内部配置等敏感信息。正则匹配 + 关键词黑名单双重过滤。

技能选择机制

LLM 驱动的技能路由

分析 Agent 接收到分析请求后，通过 LLM 判断应使用哪个技能（或技能组合）。Prompt 中包含所有技能的描述和能力边界，LLM 输出技能名称 + 参数，Agent 据此实例化对应 Skill 对象执行。

易错点：技能选择错误

LLM 可能将"估值分析"误判为"财务分析"，导致执行错误技能。解决方案：技能描述中增加明确的触发条件和边界说明；选择后增加校验步骤，确认技能与意图匹配。

技能组合策略

多技能协同

综合分析技能可编排其他技能的输出作为输入，形成技能组合链。例如：综合分析 = 财务分析 + 估值分析 + 风险分析 → 权重融合 → 综合结论。通过 BaseSkill 的统一接口实现技能间无缝衔接。

易错点：技能间数据传递

前序技能输出格式变化可能导致后续技能解析失败。解决方案：BaseSkill 强制统一输出 Schema，所有技能输出必须包含 status + data + summary 三个标准字段。

核心难点与易错点总结 面试重点

六大核心难点

1. 意图识别准确性

LLM 输出格式不稳定、多意图拆分依赖关系判断、模糊意图的边界划分。两阶段 Pipeline + 结构化约束 + fallback 机制三重保障。

2. 反爬对抗持续性

金融网站反爬手段持续升级，stealth 插件需持续维护。三层防御体系（行为/特征/频率）+ BrowserManager 统一管理 + 异常截图排查。

3. Prompt 链稳定性

多步 Prompt 串联时中间输出漂移导致后续解析失败。模板方法模式 + 每步格式校验 + 重试降级策略 + 统一输出 Schema。

4. 记忆一致性

三层记忆信息冲突、TTL/FIFO 边界条件、向量索引更新遗漏。优先级策略 + LFU+FIFO 混合淘汰 + 写入后立即验证索引。

5. 容错与降级

重试风暴、状态回滚不完整、部分降级用户体验差。全局重试协调器 + State 快照整体回滚 + 逐项 status 标注。

6. 安全与权限

SSRF 防御、URL 白名单绕过、API Key 泄露、LLM 输出越权。重定向校验 + IP 禁连 + 环境变量注入 + 输出过滤。
