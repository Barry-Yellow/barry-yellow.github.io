AG-Repair · 01
项目概述 & 核心问题
01 / 09
项目定位
解决多智能体系统在长轨迹执行中出现的失效问题，在最小干预、最大复用的前提下完成自动化修复与增量恢复。
❌ 传统方法
发现错误后全量重跑，假设错误点之后所有内容失效
✅ AG-Repair
将失败轨迹视为可分析、可切分、可局部修补的结构化过程
技术栈
Python LLM API 图算法 SQLite JSONL Checkpoint 合成数据 故障注入
四大设计目标
  • 精准定位 — 找出真正需要修复的关键错误点
  • 量化影响 — 区分可复用节点 vs 必须修补节点
  • 成本感知 — 优先选择代价最小的修复动作
  • 增量恢复 — 只重放必要部分，一致性校验
整体修复流程（7步闭环）

AG-Repair · 03
核心算法：根因定位 · 污染传播 · 修复动作
03 / 09
根因评分算法
S_root(v) = α·A(v) + β·I(v) + γ·D(v) + δ·R(v)
A(v)异常强度 — 节点上观测到的错误证据强弱
I(v)影响强度 — 向后能到达多少潜在受影响节点
D(v)偏离程度 — 与任务目标/约束之间的偏差
R(v)修复收益 — 修复该节点的潜在恢复价值
策略：硬信号优先（A,I权重高）→ 软信号补充
污染传播 & 复用评分
P(v) = max_{u∈Pred(v)} P(u) · w(u,v) · λ
λ=0.85 衰减系数，前向传播
S_reuse = 0.5·S_correct + 0.3·(1-P) + 0.2·S_compat
节点分类阈值：Clean≥80 / Partial 60-80 / Invalid<60
5类错误模式
evidence_fabrication tool_call_error context_drop reasoning_bug plan_bug

合成数据 & 故障注入
早期阶段真实失败日志稀缺，采用合成数据优先策略，通过可控错误注入生成训练样本。
任务类型：web_research tool_chain code_repair
注入器：5类错误模式，精确标注根因/污染/安全节点
Train 60%
Dev 20%
Test 20%
增量重执行三要素
  • Checkpoint 恢复 — 从最靠近根因的 checkpoint 恢复，避免污染状态
  • Journal Replay — 未修改节点优先回放旧结果，不重调外部环境
  • 依赖注入 — 只注入 Active Dependency Frontier，减少长上下文干扰
局部失败 → 动作升级（最多2轮），避免无限重试

FinAgent · 04
技术难点 & 项目总结
08 / 09
核心技术难点
LLM 输出格式不稳定
strip() + 正则提取 JSON + json.loads() + fallback 关键词匹配
动态渲染页面采集
Playwright 智能等待（networkidle），BrowserManager 单例管理生命周期
多意图依赖顺序
sub_intents 按依赖排序串行执行，前序输出自动注入后续 context
记忆一致性冲突
优先级：短期 > 长期 > RAG，LFU+FIFO 混合淘汰策略
项目亮点总结
  • LangGraph 声明式编排，StateGraph 管理复杂多 Agent 流转
  • 两阶段意图识别降低 LLM 调用成本，支持多意图串行
  • Agentic RAG 混合检索 + RRF 融合，优于传统单路向量检索
  • 三层记忆实现跨会话知识积累，优先级覆盖保证一致性
  • 三层反爬（行为/特征/频率）应对金融网站反检测
  • 四层容错保证系统鲁棒性，部分降级而非全量失败
  • 9大分析技能 + 渐进式 Prompt 链，模板方法保证输出一致性
安全设计
  • 域名白名单防 SSRF，跟随重定向后二次校验
  • API Key 环境变量注入，日志自动脱敏
  • LLM 输出后处理过滤，防止泄露系统 Prompt
